# 窮鬼勇者 Visual System v1

## Codex 工作前硬規則

任何 UI、角色、背景、地圖、圖示、套裝、動畫或首頁版面改動，動手前都必須先閱讀本文件。若需求與本文件衝突，先以本文件為準；若使用者當下明確更新規則，必須同步修改本文件再改程式。

執行順序：

1. 先確認本文件中的相關段落。
2. 再讀實際程式與資產引用。
3. 改動後將新增規則寫回本文件。
4. 最後 build 驗證並 commit/push。

## 美術核心

**日系魔法學院 × 溫暖財務冒險。** 畫面要讓記帳像一次輕巧的冒險行動，但財務數字永遠比裝飾更清楚。

### 三個視覺關鍵字

- 溫暖：奶油紙張、日光金、低刺激陰影
- 魔法：星術紫只負責品牌與稀有內容
- 清楚：深靛文字、薄荷安全色、珊瑚警示色各司其職

## 色彩規則

| 角色 | Token | 色值 | 用途 |
|---|---|---:|---|
| 品牌主色 | `--color-brand-500` | `#7161B8` | 目前頁籤、主要品牌標記 |
| 品牌深色 | `--color-brand-700` | `#433A78` | 深色標題、焦點邊框 |
| 獎勵金 | `--color-reward-500` | `#E7AD36` | 黃星、獎勵、限定亮點 |
| 安全薄荷 | `--color-safe-500` | `#2FAF9F` | 預算安全、完成狀態 |
| 警示珊瑚 | `--color-danger-500` | `#D96573` | 超支、失敗、破壞性操作 |
| 主文字 | `--color-ink-900` | `#27304A` | 主要數字與文字 |
| 次文字 | `--color-ink-500` | `#77738D` | 說明文字 |
| 奶油底 | `--color-surface-base` | `#FFF8EC` | App 底色 |

規則：一個元件最多一個強調色；粉紅不再作為通用裝飾色，只用於警示或特定收藏品。

## UI 形狀與材質

- 面板圓角：24px；小元件：14px；膠囊：999px。
- 面板使用暖白半透明材質，不使用純白冷玻璃。
- 一般面板只用一層陰影；重要 CTA 可以增加金色內光。
- 每頁只能有一個主 CTA。
- 背景負責世界觀，UI 負責資訊；UI 不再和背景競爭細節。

## 圖示系統

- 導航／功能圖示：圓潤 2D、2.5px 等效描邊、單色為主。
- 貨幣／獎勵物件：2.5D 道具，可有高光與材質。
- 禁止混用 emoji 作正式圖示。
- 商店、任務與獎勵列的貨幣數量必須使用正式資源圖示加數字，不寫成「黃色星星 4」「紫色星星 x1」這類純文字價格。
- 資源摘要卡若同時顯示四種貨幣，優先只顯示圖示與數字；貨幣名稱可放在 title/aria-label，不必常駐佔版。
- 同組輸出統一 64×64 邏輯畫布及 8px 安全邊距；貨幣 PNG 可保留 256×256 原始輸出，但圖形必須在 18-32px UI 尺寸下仍清楚。
- 狀態只使用 `default / active / disabled` 三種；active 由底板與顏色共同表達。
- 角色換裝入口不建議命名為「衣櫃」；主頁角色旁的入口應優先使用「造型」或「套裝」這類遊戲語彙。
- 造型入口圖示必須一眼可辨，優先使用衣架、衣服加星星、魔法試衣鏡；避免抽象櫃門或門板圖形，縮小後會像一般功能按鈕。

## 完整主題套裝收藏系統

每份獎勵是一套完整角色演出，不拆成可自由混搭的紙娃娃零件。收藏價值來自新鮮感，而不是只替換上衣顏色。

### 每套必須獨特的九個面向

1. 髮型與髮飾：長度、輪廓、綁法或頭飾不得沿用同一樣板。
2. 臉部造型：保留主角辨識度，但改變妝感、眼神、表情與主題細節。
3. 服裝輪廓：不只換色，剪裁、重心與材質都要有差異。
4. 隨身物：一至二件有故事性的物件，必須由角色穿戴或手持，不得漂浮在旁邊。
5. 專屬背景：每套都有獨立場景，光源、色溫與套裝一致。
6. 場景擺飾：只留二至三個必要視覺線索，不做商品展示架式堆疊。
7. 場景呼應：角色手持物、視線、身體朝向、腳底位置需與背景元素形成關係；不能只是把角色貼在背景前。
8. 專屬動作：動作要與場景互動，不得全部套裝共用呼吸、眨眼或同一轉圈。
9. UI 氣氛：主 CTA、安全色與微光可隨主題調整，但不犧牲數字可讀性；避免制式白底玻璃卡，HUD 材質要跟場景同一套語氣。

### 節日與季節限定套裝

節日套裝可以做，而且應該成為長期收藏與回流活動的核心，但不能只是把節日道具貼在一般服裝上。

- 每個節日套裝都要包含：節日髮型/髮色、服裝輪廓、手持物或大型配件、專屬背景、場景小動畫、專屬動作與活動故事。
- 節日元素可以誇張可愛，但必須穿戴或手持在角色身上，例如端午的大粽子帽、香包腰飾、龍舟槳、艾草髮飾；不能漂浮在旁邊。
- 節日限定優先做同主題男女雙版本；背景與活動名稱一致，但男女角色的髮型、剪影、表情、動作各自設計。
- 節日套裝可搭配節日地圖覆寫、限定怪物、任務與較稀有獎勵，形成完整活動包。
- 完整節日套裝不進普通商店、不進一般直購架、不進常駐扭蛋池；只能透過活動任務線、節日週獎勵或明確標示的復刻活動取得。
- 不必一次補齊全年節日，優先順序建議：春節、端午、中秋、聖誕/跨年，再補其他台灣國定假日。

#### 端午套裝方向

端午套裝應偏「夏初河岸祭典 × 龍舟 × 粽子魔法」，不要只是綠色衣服。

- 服裝：短版祭典外套或輕便學院運動祭服，搭配艾草、五色繩、香包與水紋刺繡。
- 髮型/髮色：可改成清爽高馬尾、側辮或短髮，髮色可偏艾草綠、粽葉青、茶褐或金粽色，但仍保留主角臉部辨識。
- 大型配件：粽子造型帽可以做，但輪廓要可愛且不壓住臉；也可搭配龍舟槳、香包小袋、迷你粽子背包。
- 動作：揮槳、舉粽子、綁五色繩、向河面加油，動作要像端午活動，不共用一般待機。
- 背景：近景河岸木棧道或龍舟起點，水面有粼光、彩旗、粽葉與遠處龍舟；角色腳底要站在棧道或岸邊石階，不浮在水上。
- 小動畫：水面波光、彩旗飄動、少量粽葉/艾草飄落；不要塞滿龍舟、粽子、燈籠造成畫面雜亂。

#### 七夕套裝與活動方向

七夕主題應偏「星河學院祭 × 鵲橋 × 願望帳本」，不是一般粉紅情人節，也不是單純把愛心貼在既有服裝上。七夕可以作為夏末主檔節慶活動，優先級高於相近日期的中元小活動。

- 活動名稱方向：星橋願望祭、銀河帳本祭、鵲橋記帳夜，語氣要浪漫但仍保留財務冒險感。
- 服裝：星紗短斗篷、學院祭禮服或輕便浴衣式剪影，可加入銀線、星圖、羽毛與願望籤細節；避免大面積粉紅，主色以深靛、星術紫、月白、獎勵金與少量珊瑚點綴。
- 髮型/髮色：可用雙辮、側編髮、星形髮飾、月白挑染或深紫漸層，但要保留主角臉部辨識。
- 隨身物：願望帳本、星砂瓶、羽毛筆、鵲橋票券或小型星盤，必須手持或穿戴，不得漂浮在角色旁。
- 動作：寫下願望、舉起星砂瓶、牽引星線、在橋面行禮或回頭看星河；不可只共用一般待機。
- 背景：近景應有可踩的橋面、學院屋頂或祭典平台，中遠景可放銀河、星燈、遠處鵲橋與少量願望籤；角色腳底要有清楚落點，不浮在星空上。
- 小動畫：星塵緩慢閃爍、願望籤輕擺、遠方星線流動；不使用大量愛心、花瓣或高飽和粉紅粒子。
- 活動任務：以「連續記帳、預算內留下願望金、完成每日小目標」為主；完整套裝放在活動任務線或節日週獎勵，不放進普通每日任務。

#### 中元小活動方向

中元節若接在七夕主檔後，應作為小型接續活動或素材包，不搶七夕主視覺。

- 語氣偏「夜市普渡 × 財務避邪 × 可愛民俗」，避免恐怖、血腥或驚嚇。
- 可做限定怪物、任務文案、地圖週分區覆寫或小道具，不優先做完整雙版本套裝。
- 色彩以夜靛、燈籠金、薄荷安全色與少量珊瑚警示為主，不做陰森黑紅主題。

### 主角一致性邊界

- 必須保留：基礎年齡感、Q 版比例、主色基因與整體人格。
- 必須改變：髮型輪廓、主表情、服裝外輪廓、隨身物與動作語彙。
- 不使用角色後方的白霧展示框；角色應直接站在專屬背景裡。
- 腳底必須有明確場景落點，頭頂不貼 HUD，完整動作不得進入玩家資訊卡。

### 資產契約

- 正式版只保留透明 Animated WebP，原始影片不進入前端專案。
- 角色、服裝、髮型與隨身物合成同一動畫資產，避免動作脫節。
- 每套動畫需檢查正面、側面與動作最大幅度幀。
- 每套獨立設定腳底基準點與顯示尺寸，不共用一組 `top/width`。

### 角色動畫製作流程

每套角色都要走同一條正式流程，不能用靜態圖假裝動畫。

1. 角色主圖
   - 先生成完整角色：髮型、髮色、服裝、手持物、配件全部畫在同一張圖中。
   - 角色必須是全身可見，腳底完整，沒有裁切。
   - 主角身份要一致：紫色眼睛、Q 版圓臉、柔和五官與目前角色的年齡感。
   - 每套必須不同：髮型、髮色、服裝剪影、表情、隨身物與站姿不得只是換色。
   - 若場景是下雨，角色必須撐傘或穿雨具；若沒有傘，場景應改成雨後，而不是主動下雨。

2. 動畫生成
   - 使用角色主圖做 image-to-video，不從多張獨立圖片硬湊幀。
   - 動作必須符合場景，例如夏日可以轉圈、眺望、展示泳圈；祭典可以開扇、行禮、接花瓣；雨夜可以撐傘、翻帳本、提燈。
   - 鏡頭固定，角色不能大幅縮放或漂移到畫面外。
   - 手持物必須全程存在，不可在部分幀消失。
   - 動作幅度要有生命感，但不能讓腳底離開地面。

3. 去背與轉檔
   - 優先要求乾淨純色背景；避免綠幕碰到書本、薄荷衣服、雨衣等綠色物件。
   - 若角色有綠色或藍綠道具，改用洋紅色背景 `#ff00ff` 或其他不會撞色的純色。
   - 去背後必須逐幀檢查手持物邊緣，尤其是書本、傘、髮絲、透明雨衣與發光道具。
   - 禁止保留綠邊、黑邊、白底、棋盤格假透明。
   - 正式輸出為透明 Animated WebP，不提交原始 MP4、臨時 PNG 序列或失敗版本。

4. WebP 規格
   - 角色動畫畫布：`360 × 480`。
   - 建議幀數：`40-50` 幀。
   - 建議幀時間：`100ms`。
   - 角色腳底需落在畫布下方穩定區域，不可每幀上下飄。
   - 角色在畫布內應有一致安全邊界：頭頂不貼邊，腳底不裁切，手持物不出界。
   - 檔案大小目標：一般套裝約 `500KB-1.5MB`；超過 `2MB` 需重新壓縮或縮短幀數。

### 角色尺寸與首頁落點

角色在首頁不是單純置中，而是要像站在場景裡。

- 每套都要獨立設定 `top / width / max-width / shadow`，不得共用預設值。
- 角色視覺大小要和目前標準套裝接近；不能一套像小模型、一套像半身大頭。
- 檢查標準：腳底落點、頭頂距離 HUD、手持物最大幅度、下方行動面板遮擋。
- 首頁所有套裝的可見腳底 baseline 以「預設女主」為基準；真機/PWA 截圖優先於桌面瀏覽器手機模擬，手機模擬只能做粗略檢查。已驗收的目前落點：預設女主 `top:260px width:58vw max-width:228px`，新套裝以角色資產透明像素的可見腳底對齊此 baseline，而不是對齊透明畫布底部。
- 角色腳底應踩在背景中的明確平面，例如階梯平台、沙灘、石板路、學院廣場。
- 背景物若是主體建築，比例要合理：鳥居、校門、拱門等大型結構應大於角色，不應像遠方小裝飾。
- 不能靠把角色越縮越小來製造巨大背景；應重新生成或升級背景解析度，讓場景本身有正確透視。
- 手機與桌機要分別檢查，因為真機瀏覽器高度、safe area 和 app 框高度不同。
- 手機底部行動面板與 dock 必須保留安全距離，但不能高到像懸浮在畫面中央。

### 背景製作流程

背景是套裝的一部分，不是角色後面的裝飾圖。

1. 背景主圖
   - 每套有專屬背景，尺寸建議至少 `1536 × 2048` 或等效高解析 9:16。
   - 背景不得包含角色、文字、數字標記、UI、logo 或浮水印。
   - 必須預留角色站位，但站位不是空白框；要有可踩的地面、平台或道路。
   - 透視要符合拍照感：角色像站在現場，背景前景元素能包圍角色。
   - 場景主體要靠近並有尺度，例如鳥居像拍照框、海灘有近景沙地、雨巷有近景石板與店燈。

2. 背景清晰度
   - 不使用低解析背景硬放大到糊。
   - 若構圖方向正確但解析度不夠，先做 upscale，再壓成前端 WebP。
   - 背景 WebP 目標大小約 `150KB-600KB`；若超過 `1MB` 需檢查壓縮品質與裁切。

3. 場景小動畫
   - 背景圖本身可以是靜態，動態由 CSS 或小型透明 WebP 疊層完成。
   - 動態元素要符合場景：櫻花落瓣、夏日雲層與水光、雨夜燈光與雨痕、學院星塵與紙張。
   - 雨後偵探若角色沒有雨具，主頁與收藏縮圖都只能使用雨後濕地反光、燈光呼吸、微弱水痕 shimmer；不得使用正在落下的雨線。
   - 小動畫只服務氛圍，不得搶角色主動作。
   - 不要堆太多擺飾；每套場景只保留二至三個必要線索。

4. 場景與角色一致性
   - 角色道具要和背景互相解釋，例如泳圈對應海灘、扇子對應祭典、傘對應雨夜。
   - 光源方向與色溫要一致；夜景角色不能像白天棚拍，海灘角色不能套夜色陰影。
   - 如果角色動作是看遠方，背景要有可看的方向；如果角色行禮，背景要像儀式或入口場景。

### 套裝驗收清單

每套進專案前至少檢查以下項目：

- 角色是否是透明 Animated WebP，且不是靜態 PNG 替代品。
- 角色是否符合主角臉，但髮型、髮色、服裝與動作足夠新鮮。
- 動畫中手持物、髮飾、包包、傘、書本是否全程存在。
- 邊緣是否乾淨，沒有綠幕殘影、黑邊、白底或棋盤格。
- 角色大小是否接近既有標準套裝，腳底是否踩在正確地面。
- 背景是否有近景、中景、遠景，角色是否真的站進場景。
- 場景主體比例是否合理，大型建築不能小於角色。
- 手機和桌機是否都能看到相同的主體關係。
- 下方行動面板是否不遮腳、不黏 dock、不離角色過遠。
- UI 材質是否跟場景同一語氣，但財務數字仍清楚。

### 首頁 HUD 與場景空間

- 預算資訊只在下方行動面板出現一次：`今日消費 / 今日可用預算`。
- 上方 HUD 只放角色身份、稱號、成長入口與收藏貨幣，不重複預算進度條。
- 上方 HUD 身份區必須保留頭像展示位，頭像框、玩家身份與收藏貨幣共用同一個狀態板；不得只顯示文字名稱。
- 上方 HUD 必須保留一條輕量 EXP 進度線，任務、月度挑戰與活動獎勵都會回饋到這裡；EXP 不能做成新的大卡片。
- 身份與貨幣可收斂為同一個場景式狀態板；材質跟隨套裝，如夏天用清透水光、夜櫻用深色燈籠光。
- 下方行動面板至少要有三個有意義的狀態：今日消費、今日攻擊力、今日可用預算；避免左右資訊之間留下像缺資料的空白。
- 若主頁採「造型縮圖式展示舞台」方向，角色腳底與魔法圈/地面落點優先完整露出；下方行動面板應改成薄型 HUD，不得用大卡片吃掉角色腳底、魔法圈或主要地面舞台。
- 角色腳下不放姓名貨幣大卡，中央區域優先留給角色動作、腳底落點與背景景深。
- 記帳步驟只在新手教學中出現，不常駐於首頁。
- 主頁角色旁的換裝入口應使用「造型」或「套裝」文案，並搭配衣架、衣服星星或魔法試衣鏡圖示；不使用辨識度低的抽象衣櫃門。

### 主頁特效商品分層

主頁特效可以成為商店與活動兌換商品，但必須先做成可組合、可預覽、可控尺寸的分層系統，而不是把整張背景重畫進同一張圖。

- 背景圖可以保持靜態；動態由透明 Animated WebP、CSS 動畫或兩者混合的疊層完成。正式高價商品可用 Higgsfield 產出素材，但前端仍只接收可定位的透明疊層。
- 地面特效、背景氛圍特效、記帳成功特效是不同商品槽位；不能把三者綁死成一張不可拆的整圖。
- 每個地面特效至少要定義三段行為：`groundIntro`（裝上或進首頁短暫演出）、`groundIdle`（平常低頻待機）、`groundSuccessSync`（記帳成功時跟著亮一下）。
- 每個記帳成功特效至少拆成 `successBurstBack` 與 `successBurstFront`；前者在角色後方產生光束或爆光，後者在角色前方產生金幣、星星、票券等獎勵粒子。
- 記帳成功特效的主要演出應由透明 Animated WebP 主導；CSS 只負責定位、觸發、淡入淡出與低成本備援，避免主視覺看起來像規則的工程光束。
- 首頁常駐的地面特效不得過大；以學院法陣為標準，待機圈只做「角色腳下舞台」，寬度約角色可視寬度的 2.1-2.4 倍，並保留底部 dock、未來寵物與小擺飾空間。
- 記帳成功特效只能在記帳完成、返回主頁或商店預覽重播時觸發；不得在首頁永遠爆光。
- 每個背景或套裝必須有自己的 stage profile，至少包含腳底 baseline、地面特效中心點、地面特效最大寬度與可相容地面類型。沒有明確空地或平台的背景不得硬套大型法陣。
- 學院法陣第一版只視為學院舞台範例；若要讓其他背景自由搭配，背景製作時必須預留和學院一樣清楚的可踩地面或另做該背景專用縮放設定。
- 商店預覽必須能分開看「裝上/進場」、「平常待機」、「記帳成功」三種狀態，避免玩家誤以為購買後所有特效都會同時常駐。

### 手機 Safe Area 與滿版背景

- 場景背景必須延伸到整個螢幕，包括 iOS 狀態列、瀏海與底部 safe area；不得讓狀態列後方露出純色空白。
- UI 內容才需要避開 safe area：首頁 HUD、返回列、設定頁標題、底部 dock 等操作層使用 `env(safe-area-inset-*)` 留安全距離。
- 不得為了避開瀏海把整個 app 或背景往下推；這會造成手機與桌機看到的場景不一致。
- PWA / iOS 狀態列應使用透明或半透明語氣，讓場景背景成為狀態列背後的底色。

### 首頁版位鎖定

首頁 HUD、下方行動面板與底部 dock 是制式操作層，不跟著套裝任意位移。

- 上方身份/貨幣 HUD 固定在相同外距與高度區域。
- 下方行動面板固定貼近 dock 上方，保留安全距離，但不得因套裝而忽高忽低。
- 底部 dock 固定；不得為單一套裝調整 dock 位置。
- 套裝只能調整：背景裁切、角色尺寸、角色腳底落點、角色陰影、場景粒子與主 CTA 色調。
- 禁止在 `.academy-screen--主題` 內覆寫 `.academy-home-content` 或 `.academy-attack-panel` 的 `padding-bottom / margin-bottom`，除非是全站版位規則同步更新。
- 驗收時至少比較 `academy / summer / sakura / rainy` 四種主題，確認上下資訊卡位置一致。

## 地圖與遠征路線

地圖頁不是資料表，也不是白卡片上擺節點；它是每日與每週遠征路線。月度總結、分類統計與長期消費回顧歸入獨立報表/月報，不塞在地圖主流程裡。

### 地圖底層

- 底層應是中性的遠征地圖材質，例如羊皮紙、魔法路線圖、星術地圖或高空學院地景；不得使用與分區卡互相打架的完整場景圖。
- 分區可以有自己的插畫，但需要融入同一張月度路線，而不是像四張商品卡疊在舊背景上。
- 分區背景可保留，但白霧遮罩不得厚到看不出地點差異。
- 月初區域可以明亮、月中區域可以誘惑或迷霧、月底 Boss 區域要有壓力與終點感。
- 地圖背景不應全年固定不變。基本規則是「月度基底 + 週分區 + 節慶覆寫」：
  - 一般月份可使用四週固定節奏：學院入口、市集、森林/迷霧、月底魔王城。
  - 若當週包含台灣重要節日或國定假日，該週分區圖可被節慶版覆寫，例如春節、端午、中秋、聖誕、跨年。
  - 節慶覆寫只改該週場景、怪物/獎勵語氣與少量 UI 氣氛，不改整個月的操作結構。
  - 每月月底週必須有更強壓力與 Boss 終點感，不能和普通週同等視覺重量。
- 地圖素材不應變成每月從零畫四張完整大圖。優先採用可控素材量：
  - 一張通用月度底圖或路線材質。
  - 四個可重複週分區模板：第 1 週學院入口、第 2 週魔法市集、第 3 週迷霧/壓力區、第 4 週月底 Boss 城。
  - 節日週才覆寫該週分區，不整月重畫。
  - 年度新鮮感來自節日覆寫、Boss 變體、怪物與獎勵，不靠每月大量新增背景。
- 地圖頁改版必須四週一起驗收；不得只把月底 Boss 區做成完整場景，卻讓前幾週維持白卡或空白格狀節點。
- 週分區可以各有氣氛，但節點、路線、日期與區塊材質必須共用同一套遠征地圖語言，不能讓某一週看起來像不同頁面或不同遊戲。
- 地圖上方只放本週戰報，呈現今日位置、本週記帳天數與週任務獎勵狀態；本月總消費、日均、S/A/B/C 統計等月度資訊移到月報。
- 本週戰報應像貼在地圖上的遠征戰報、羊皮紙或告示板，而不是一般白色報表卡；資訊可以保留精簡，但材質要融入地圖頁。
- 戰報不得只是把白卡改成暖白卡。需要有明確紙張/告示結構，例如裝訂邊、印章、墨跡分隔、小型統計欄；統計格也不能維持泛白報表框。
- 地圖頂部不重複顯示戰報內已有的完成天數；`0/7 天` 這類進度只能保留一個主要位置，優先放在本週戰報。
- 進入地圖頁時應自動捲到今天所在的當週區塊，讓地圖作為「本週路線、補登、週任務獎勵」工具，而不是每次從月初開始。
- 地圖圖例不常駐佔用版面。節點說明預設收進右上角 `?` 說明入口；第一次進地圖的教學可以等功能穩定後再做，不提前綁死流程。

### 節點與日期

節點必須是遊戲地圖標記，不使用通用功能圖示充當日期。

- 未來：灰白石碑或未啟動路標。
- 未記：空白帳本頁或封印中的小節點。
- 已記但未擊殺：裂盾、爪痕或待討伐標記。
- 已擊殺：星章、小旗或亮起的勝利徽記。
- 今日：發光足跡、定位旗或正在閃爍的路標。
- 週末 Boss：小型怪物門、盾牌徽記或紅色挑戰標記。
- 月底 Boss：大型城門、魔王封印或壓迫感更強的終點節點。

日期呈現固定為：上方狀態圖案、中間日期數字、下方狀態/日字；Boss 節點可以放大，但仍要能清楚辨識日期。

### 補登規則

- 補登只修正歷史消費與統計。
- 補登不重新發放當日戰鬥獎勵、扭蛋券或完整任務收益。
- 若未來要給補登鼓勵，只能給比當天正常記帳更低的補償，例如少量經驗或無貨幣紀錄徽章。

## 男女性別與套裝策略

- 玩家首次進入遊戲必須選擇性別與名稱，避免預設只看到女生。
- 對使用者文案優先稱為「主角外觀」或「主角版本」，避免像填資料表一樣強調生理性別；資料欄位可繼續使用 `avatarGender`。
- 初始登錄與設定頁的主角外觀選擇是偏好切換，不是套裝預覽；若正式男女主角資產尚未完整，不使用舊紙娃娃圖片作大圖展示。
- 預設學院套裝必須同時有男生與女生版本；兩者可共用學院背景，但角色服裝、髮型與動作要各自完整。
- 季節/活動套裝優先做「同主題雙版本」：同一活動名稱下有男生版與女生版，背景與事件一致，角色演出不同。
- 若只製作單性別限定套裝，衣櫃必須清楚標示，不要讓另一性別 fallback 到不合適的角色。
- 衣櫃列表必須依當前主角外觀過濾不支援的套裝；若目前裝備不支援當前外觀，預覽與可裝備狀態要 fallback 到第一個已擁有且支援的套裝。
- 男生不能只是女生服裝改髮；需要獨立剪影、髮型、表情、姿勢與隨身物，但仍維持同一遊戲畫風。
- 男版節日套裝不得直接使用女版髮夾或偏女版髮飾。可改用星形胸針、肩飾、領結、髮束、帽飾或其他中性／男版設計；七夕男版髮夾問題列為後續重畫 TODO。

## 設定與帳本管理

- 設定頁負責系統與身份偏好：暱稱、主角外觀、資料與同步、聲音/音樂/震動、教學與隱私。
- 記帳分類屬於記帳流程，不放在設定主頁；玩家應能在記帳當下新增或管理分類，避免離開戰鬥/記帳流程。
- 主角外觀不等於套裝衣櫃。設定頁只提供簡化切換；套裝收藏、套裝預覽與取得來源仍從主頁「造型」入口進入。
- 聲音設定需預留三類：背景音樂、操作音效、震動回饋。即使音樂尚未實作，也應在設定語意上和音效分開。

## 任務、成就與獎勵

- 任務頁分工：每日任務負責今天的小目標；每週獎勵連接地圖週進度；成就負責永久長期累積；活動負責節日與期間限定獎勵。
- 地圖每週區塊右上角標籤固定作為「任務獎勵」狀態，不再只是區域名稱。條件預設為該週每一天都有記帳；未完成顯示進度，可領顯示可領取，領完顯示已領。
- 任務頁與商店頁不能只是一串白色卡片；頁面必須有專屬背景語氣、輕量動態氛圍與有色材質分層。
- 每日任務可以固定保留基礎任務，但必須有日期固定的輪替池，避免每天看到完全相同清單。
- 貨幣、票券等獎勵不得只用文字寫「黃色星星 2」；必須使用正式獎勵圖示搭配數字。EXP 可以用文字或輕量膠囊呈現。
- 完整套裝不放在普通每日任務裡，應作為節日週、活動任務線、長期成就或月底 Boss 等較稀有獎勵。

## 戰鬥與記帳攻擊

- 戰鬥頁的主目的仍是快速記帳，怪物演出負責給回饋；不得讓純戰鬥資訊壓過金額輸入。
- 版面優先順序：今日討伐頂列、薄型狀態、怪物舞台、記帳攻擊面板、今日明細摘要。
- 戰鬥頁的怪物必須是上半部主視覺；今日消費、攻擊力、HP 等資訊應以薄型 HUD 或舞台 overlay 呈現，不再堆成多張大型白卡壓縮怪物。
- 怪物名稱、血條、首領標籤與掉落提示應整合進怪物舞台 overlay；避免在舞台上方另放完整 HP 卡。
- 已確認方向為「卡片退場，怪物當主角」：怪物舞台內不得再出現完整資訊卡；怪物名、首領標籤、HP 與掉落提示只能作為貼邊文字、細血條或小徽章，且不能遮住怪物主體。
- 戰鬥頁頂部只保留返回、標題、剩餘預算與今日明細入口；今日消費、攻擊力、HP 不再獨立成一條卡片列，改由怪物舞台 overlay 承載。
- 戰鬥頁若下半部出現明顯空白，優先擴大怪物舞台與壓縮記帳攻擊卡；不可用新增資訊卡填空。
- 今日紀錄在戰鬥頁只做入口，不佔主內容高度；入口放在頂列右側與剩餘預算同層，點擊後以 drawer / modal 展開明細。明細面板可承載單筆修改、刪除、分類修正等管理操作，避免越記越多時壓縮怪物舞台與記帳攻擊面板。
- 記帳攻擊面板只能有一個主要 CTA：`記帳攻擊`。
- 戰鬥頁在標準手機與 PWA 高度下必須能完成「選分類 → 輸入金額 → 送出」，且輸入金額後不需要下滑才看得到主要 CTA；底部 dock、safe area 或鍵盤不得遮住主要 CTA，內容區必要時必須可捲動。
- 加總功能若只支援 `+`，文案必須稱為「多筆加總」，不得稱為完整計算器。
- 攻擊特效商品的正式顯示位置是戰鬥頁攻擊動畫；商店預覽必須能對應到這裡。
- 擊敗回饋要顯示掉落預告、任務進度或結算徽章提示，但正式獎勵發放仍遵守每日結算與既有規則。
- 怪物與背景可以分階段重畫；在版面規格未穩定前，不先大量重做怪物資產。
- 戰鬥頁若暫用主頁背景，必須加上戰鬥專用舞台層，例如魔法陣、封印線、聚光與暗角，避免看起來只是主頁換了怪物。
- 未來專屬戰鬥背景要偏帳本廣場、任務告示板與封印魔法陣；怪物風格是可愛財務壓力，不做恐怖或血腥。

## 主頁記帳入口與展示空間

- `立即記帳 / 立即戰鬥` 是高頻入口，功能必須保留；但不必永遠使用大型底部卡片呈現。
- 若要提升角色、寵物、地面特效、背景特效、擺飾等獎勵展示價值，可以把記帳入口改成底部 dock 中央圓形主按鈕、浮動 FAB 或更輕量的短條入口，讓角色舞台取得更多高度。
- 底部 dock 確認方向：`今日 / 地圖 / 記帳 / 任務 / 選單`。其中 `記帳` 是中央凸起主行動，直接進入戰鬥/記帳攻擊；`選單` 使用四宮格圖示，收納 `商店 / 公會 / 月報 / 設定` 等次要入口。主頁右側 `造型` 入口維持原位置，不收進選單。主頁 HUD 不再放設定齒輪，設定統一從選單開啟。
- 選單展開必須偏遊戲快捷列，而不是網頁式大型 bottom sheet：點擊右下 `選單` 後，選項按鈕從 dock 上方向上排列展開，可輕量遮罩但不得用大卡片覆蓋角色主畫面。`月報` 必須使用報表 / 圖表 icon，避免與 `地圖` icon 混淆。
- `補給` 作為底部與選單入口名稱時改稱 `商店`；補給券、補給池等遊戲物件名稱可保留。
- 主頁底部大型 `立即記帳` 卡退場後，角色舞台要放大；記帳主入口由 dock 中央 `記帳` 承擔。除 `今日` 主頁外，其他專注頁若已提供左上返回，就不常駐底部 dock。
- 主頁背景必須延伸到手機狀態列 / safe area 後方，HUD 自己避開瀏海；不可讓頂部露出與場景無關的純色空白條。
- 真機狀態列 / PWA 外層 chrome 需要同步目前頁面或套裝的頂部主色與漸層；不可只設定畫面內背景或單一 `theme-color`，否則 iOS 會露出全站預設色或看起來像一條空白色帶。
- 調整主頁記帳入口時，必須同時確認角色腳底基準、底部 dock、安全區、地面特效與背景特效不互相遮擋。

### 主頁特效分層與商品化

主頁特效可以作為商店、活動與成就的可兌換收藏，但必須先符合展示系統規格，不得把單次 demo 寫死在某個套裝裡。

- 主頁展示層固定分為三類：`backgroundAura` 背景氛圍、`groundEffect` 地面舞台、`successEffect` 記帳成功觸發特效。未來寵物與小擺飾可另成分類，但不得和這三類混用欄位。
- 每類特效都必須有穩定 id、中文名稱、出現位置與適用主題說明，讓商店、收藏與首頁可以引用同一份定義。
- `groundEffect` 與 `successEffect` 不得因主題自動 fallback 出現；玩家沒有裝備時，學院地板就是空石板地，記帳成功也不顯示商品級爆光。若新手預設有法陣，必須是初始已擁有且已裝備的免費商品，不是背景自帶。
- 驗證期可以讓新手初始裝備免費 `starter_magic_circle` 與 `coin_spark_burst`，但必須寫入 `equipped` 與收藏資料；不得在 render 階段用 theme fallback 偷偷顯示。
- 商店必須能試看主頁特效商品，試看畫面應使用同一套首頁展示層，讓玩家看見「裝上後才出現」的效果，而不是靜態商品圖。
- 商店主頁特效試看必須接近首頁實景：使用實際背景裁切、正式角色圖與同一條腳底 baseline；不得用抽象漸層牆、網格舞台或縮圖展示盒代替，否則無法判斷商品裝上後的真實效果。
- 背景氛圍是低頻慢動，例如星塵、花瓣、雨後水光、燈光呼吸；不得永遠高亮閃爍或遮住角色臉與 HUD。
- 地面舞台必須跟角色腳底 baseline 綁定，而不是跟透明角色畫布底部綁定。所有地面光圈、魔法陣、水面 shimmer 都以預設女主可見腳底落點為校正基準。
- 地面舞台位置驗證必須以首頁正式畫面為準，商店試看只能輔助確認。實作時要同時檢查角色可見腳底、法陣視覺中心、底部 dock 遮擋與手機 viewport；不得只因試看 modal 置中就視為完成。
- 主頁角色與地面舞台是一組舞台錨點：若角色整體站位調整，`--home-stage-baseline` 必須同步調整，避免角色腳底與地面特效分離。學院主頁不得讓角色與法陣長期漂在畫面中段，底部 dock 上方需保留舞台落地感。
- 地面特效需分前後景：地板法陣、腳底光與內圈在角色後方或腳下；升起光圈、掃光與記帳成功爆光可在角色前景短暫出現，但不得蓋住臉或 HUD。商店試看與首頁必須使用同一套前後景分層。
- 地面舞台的常駐狀態只能是低亮度 idle，不得長時間高亮搶過角色。外圈不得貼近底部 dock；升起光圈不得停留或循環切過角色腰身，只能在進首頁、換裝/重播或成功觸發時短暫由腳邊往上淡出。
- 地面舞台商品必須是完整可辨識的地板視覺物件，不只是小型 glow 或陰影。學院套裝的基準款是完整星砂魔法陣：外圈、內圈、幾何陣線、符文刻度與腳底光環一起構成可兌換特效。
- 地面舞台若作為商店商品驗證，不得只用淡色 CSS 線圈假裝完成；基準款法陣至少需要一張可辨識的透明主視覺資產或同等精細的向量層，再疊加 CSS 的浮光、升起光圈與粒子。
- 若背景圖本身已有法陣構圖，實作仍應把可更換、可發光、可動態調整的法陣層放在 `groundEffect`，避免玩家以為地板特效只是背景圖的一部分。
- 地面舞台的預設動畫語言是「從地板往上顯現」：淡入、升起、掃光、垂直光粒或短暫光柱。不得把旋轉作為預設表現；只有商品本身語意是旋轉法陣時才允許明顯持續旋轉。
- 學院背景若要驗證特效商品化，應優先使用無法陣的石板地面版本；法陣、光圈、浮光與觸發爆光都從疊層產生，避免背景和商品混在一起。
- 記帳成功特效只能在完成記帳、擊殺或明確預覽時短暫播放；不得常駐循環，避免把首頁變成持續爆光。
- 記帳成功若發生在戰鬥/記帳頁，必須先排入 pending 狀態，等使用者回到主頁且主頁場景完成掛載後再播放一次；不得只在送出記帳當下發 pulse，否則切回首頁時玩家會看不到商品效果。
- 成功爆光必須是明確短事件：角色 cue、腳邊爆亮、上升光束、星砂/金幣粒子同步出現，約 0.8 到 1.2 秒結束；未裝 `successEffect` 時只能播放輕量普通回饋，不得顯示商品級爆光。
- 記帳成功特效的視覺節奏應接近四格概念：平常地面可空白或低亮 idle；觸發時先讓腳底/法陣爆亮，再有一至兩道垂直或扇形光束往上，最後金幣、票券或星星從腳邊向上散開。它不是地面法陣的常駐動畫，也不能只做成中心小粒子閃一下。
- 地面舞台的升起光圈屬於裝備/重播/入場展示，不應在記帳成功時完整重播蓋過成功爆光；成功觸發可讓地面亮度同步脈衝，但主視覺應交給 `successEffect` 的光束與獎勵粒子。
- 成功爆光也需分前後景：地面爆亮、法陣強光與上升光束優先放在角色後方或腳下；金幣、票券、星星等獎勵粒子可在角色前方飛出。不得讓大面積白光或光圈蓋住角色臉部。
- 商品級成功爆光不得只依賴 CSS 線條；驗證版至少要有可引用的透明 PNG/WebP/Animated WebP 資產作為主視覺，再用 CSS 控制位置、播放與補光。
- 角色動作需獨立於地面特效存在：未裝地面特效時仍有角色 idle、進首頁小入場與記帳成功小慶祝；已裝地面特效時，角色小慶祝可同步帶動法陣亮度與前景爆光，但不得把角色動作綁死為某個法陣商品專屬。
- 商店可先開放 CSS-only 版本驗證兌換與裝備流程；正式稀有特效若需要更高辨識度，再補透明 WebP 或序列幀。
- 背景是否需要重畫依三點判斷：角色腳底沒有可踩平面、主體比例錯誤、或前景特效無法補足場景深度。若三點都不成立，優先保留背景圖並增加疊層特效。
- 雨後偵探這類沒有傘或雨衣的套裝，不得裝備正在落雨的背景氛圍；只能使用雨後濕地 shimmer、燈光呼吸、水滴殘影等雨後效果。
- 商店驗證期可以把所有主頁特效類型做成可兌換商品，但完整商店經濟、稀有度與活動投放規則需等實機確認效果不吵、不掉幀、不破壞腳底基準後再定稿。

## 衣櫃與收藏頁

衣櫃已改為完整套裝收藏，不再以頭飾、左右手、分層紙娃娃作為主要分類。

- 主入口應顯示目前套裝的大型預覽、套裝名稱、主題背景縮圖、稀有度、取得來源與是否已擁有。
- 分類建議：全部、已擁有、季節限定、節日限定、成就套裝、未解鎖。
- 套裝列表不應只是無限長雙欄卡片；當套裝變多時要用篩選、排序與系列分組。
- 每個套裝卡應顯示：角色小動畫/縮圖、背景縮圖、性別支援、主題標籤與解鎖條件。
- 套裝卡清單必須先依當前主角外觀顯示支援項目；不支援的單性別套裝不混入另一外觀列表。若未來要展示不支援套裝，必須另做清楚的不可用分區，不可讓玩家誤以為可裝備。
- 舊的獎勵物件定位預覽、頭飾/左右手分類、分層裝備提示，只有在重新啟用紙娃娃系統時才保留。
- 衣櫃可保留頭像框、稱號、特效等收藏，但要和「角色套裝」分成不同頁籤，不混在同一個裝備表格裡。
